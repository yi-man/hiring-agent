> 模拟「集团组织与法人主体」说明。名称与汇报线为演示抽象，非实时组织图。

## 总体结构

字节跳动采用 **集团总部 + 事业群/BU + 区域公司 + 职能中台** 的矩阵式协作（模拟）。员工日常归属通常是某个业务线或平台团队，同时接受职能政策约束。

## 常见部门拆解（以研发为例，模拟）

- iOS / Android / 跨端（如 Flutter/自研跨端，演示表述）

- 业务服务、中台服务、网关与稳定性

### 数据

- 产品经理、用户研究、交互与视觉设计

### 质量与效能

- QA、测试开发、质量度量

## 分子公司与关联主体（模拟说明）

候选人常问「入职签哪家公司」。演示口径：

1. **劳动合同主体** 可能因城市、业务线而不同（北京/上海/深圳等关联公司）。
2. **社保公积金缴纳地** 通常与劳动合同及办公城市相关。
3. **业务品牌 ≠ 劳动合同主体**：例如做抖音相关工作，不等于合同上一定写「抖音」二字。
4. Offer 与入职材料会写明主体；以 HR 最终文件为准。

### 演示用主体类型

- 总部/研发型主体
- 区域经营型主体
- 海外本地实体（国际化岗位）

## 跨公司协作

- 国内与海外团队通过飞书文档、会议与异步评论协作。
- 重大共享组件由中台维护，业务线通过接口与 SLA 使用。
- 组织调整较常见，候选人应关注 **岗位职责与团队目标**，而非短期组织名。

## 部门文化差异（诚实的模拟提示）

| 类型         | 节奏                     | 候选人感知             |
| ------------ | ------------------------ | ---------------------- |
| C 端内容产品 | 快、指标强、实验多       | 需要抗压与数据敏感     |
| 企业服务     | 相对重交付与客户成功     | 需要沟通与方案能力     |
| 中台基础     | 更重长期稳定性与技术深度 | 需要工程品味与耐心     |
| 商业化       | 强目标、强协同           | 需要业务理解与结果导向 |

## Agent 回答模板

> 公司按业务线与中台组织；你应聘的岗位归属 【X 团队】，日常与 【Y/Z】协作。具体汇报线以面试官与 HR 确认为准。组织名可能调整，但岗位核心职责会在 JD 与面试中说清。
